Why This Matters

Service management questions appear in virtually every DevOps and SRE interview. Candidates who explain the systemd init system at depth — unit files, targets, boot persistence — stand out significantly.

systemctl
COMMAND
Control the systemd system and service manager
1Command Overview

systemctl is the primary interface for systemd — the init system used by Ubuntu 16+, CentOS 7+, Debian 8+, RHEL 7+, Fedora, and Arch Linux. It controls the full lifecycle of services, manages boot targets, and handles system state.

In production environments it is used to: start/stop application servers, enable services to survive reboots, diagnose failures, reload configs without downtime, and manage service dependencies. In DevOps pipelines, systemctl is embedded in Ansible playbooks, Chef recipes, Terraform provisioners, and CI/CD deploy scripts.

💡
Unit Naming Tip The .service suffix is optional — systemctl start nginx equals systemctl start nginx.service. For timers and sockets, the suffix is required.
2Syntax
BASH
systemctl [OPTIONS] COMMAND [UNIT...]

  OPTIONS  — Flags that modify behavior (--now, --user, --no-pager ...)
  COMMAND  — Action: start | stop | restart | reload | status | enable ...
  UNIT     — Service, socket, timer, or mount unit  (e.g., nginx.service)

Advanced — Unit Files, Masking & Automation
BASH
# Edit unit with drop-in override (survives package upgrades!)
sudo systemctl edit nginx
# Creates: /etc/systemd/system/nginx.service.d/override.conf

# ALWAYS run daemon-reload after editing any unit file

INI — /etc/systemd/system/mlserver.service
[Unit]
Description=ML Model Inference Server (TensorFlow Serving)
After=network.target
Wants=network.target

[Service]
Type=simple
User=mluser
Group=mlgroup

systemctl status mlserver
5Real-World Use Cases
DevOps / CI-CD Deployment
  • Zero-downtime nginx reload: After pushing new app code and updating the nginx config, run systemctl reload nginx — existing connections are never dropped.
  • Ansible handler: notify: restart nginx triggers systemctl reload nginx only when config changes — idempotent and safe.
  • Health check in deploy scripts: systemctl is-active --quiet myapp || { alert_team; rollback; exit 1; } — abort pipeline if service failed to start.
Data Engineering — Airflow / Kafka
  • Airflow scheduler: sudo systemctl enable --now airflow-scheduler — scheduler survives server reboots automatically, no manual intervention.
  • Kafka broker: systemctl status kafka for health monitoring; systemctl restart kafka after changes to server.properties.
  • daemon-reload pattern: Any time you modify Airflow's unit file (add env vars, memory limits), daemon-reload is mandatory before the changes take effect.
AI / ML Infrastructure
  • TensorFlow Serving / TorchServe / Triton: All run as systemd services in production. systemctl status triton integrates into monitoring scripts and alerting pipelines.
  • GPU persistence: systemctl enable nvidia-persistenced maintains GPU state between model loads, avoiding cold-start latency spikes.
  • JupyterHub: systemctl enable --now jupyterhub — makes multi-user Jupyter available after every server restart without manual startup scripts.
Security & Hardening
  • Reduce attack surface: sudo systemctl mask avahi-daemon cups bluetooth — prevents accidental startup of unused network services.
  • Boot audit: systemctl list-unit-files --type=service --state=enabled — security baseline check; compare against approved service list.
6Interview Questions & Answers
Q1What is the difference between systemctl start and systemctl enable?
A:start activates the service immediately for the current session only — it does not survive a reboot. enable creates a symlink in the appropriate target directory (e.g., /etc/systemd/system/multi-user.target.wants/) so the service auto-starts on every boot — but does NOT start it right now. To do both at once: systemctl enable --now nginx.
Q2What is the difference between restart and reload?
A:restart performs a full stop + start cycle — the process is killed and relaunched. All existing connections are dropped. reload sends a SIGHUP signal, instructing the process to re-read its config without stopping — connections are preserved. Not all services support reload. Use reload-or-restart as a safe fallback in scripts.
Q3What is the difference between disable and mask?
A:disable removes the boot-time symlink — the service won't auto-start, but can still be started manually. mask is stronger — it symlinks the unit file to /dev/null, making it impossible to start by any means (manual, dependency, or socket activation) until unmasked. Use mask to permanently block services like snapd or bluetooth on servers.
Q4What does daemon-reload do and when is it required?
A:daemon-reload instructs the systemd manager to re-scan unit files on disk, picking up any changes. It is required whenever you: create a new unit file, modify an existing unit file, or create a drop-in override with systemctl edit. Without it, systemd continues using its in-memory copy of the old unit definition. It does NOT restart any running services.
Q5Scenario: A service is failing at boot on a production server. Walk me through your diagnosis.
A: Step 1: systemctl --failed — see all failed units.   Step 2: systemctl status <unit> -l — view status, PID, exit code, and last log lines.   Step 3: journalctl -u <unit> -b — full logs since last boot.   Step 4: systemctl list-dependencies <unit> — check dependency issues.   Step 5: systemctl cat <unit> — verify the unit file is correct.   Step 6: Fix, then daemon-reload && systemctl start <unit>.
Q6What is the difference between Type=simple, Type=forking, and Type=notify?
A:simple (default): service is considered started when ExecStart launches. forking: for daemons that fork a child and exit — systemd tracks the child (requires PIDFile=). oneshot: runs and exits (like init.d scripts). notify: process signals readiness via sd_notify() — most robust for production services, systemd waits for the "ready" signal.
Q7Scenario: You updated Nginx config. How do you apply it with zero downtime?
A:1. Test the config first: nginx -t — exits non-zero if invalid. 2. If valid: sudo systemctl reload nginx — sends SIGHUP, Nginx re-reads config, worker processes are gracefully replaced, no connections dropped. 3. Verify: systemctl status nginx — confirm still active.
Q8How does systemctl enable work under the hood?
A:enable creates a symbolic link from the unit file into the target wants directory. For example, enabling nginx creates: /etc/systemd/system/multi-user.target.wants/nginx.service → /lib/systemd/system/nginx.service. When booting into multi-user.target, systemd follows these symlinks. The WantedBy= directive in [Install] determines which directory is used.
Q9What is the purpose of systemctl edit vs directly editing the unit file?
A:Directly editing /lib/systemd/system/nginx.service is risky — package upgrades overwrite it. systemctl edit creates a drop-in override at /etc/systemd/system/nginx.service.d/override.conf. This is merged with the original and survives upgrades. Only specify the directives you want to override. Use systemctl edit --full to replace the entire unit file.
Q10What are systemd targets and how do they relate to runlevels?
A:Targets group units together — similar to SysV runlevels but more flexible. Mapping: runlevel 0 = poweroff.target, runlevel 1 = rescue.target, runlevel 3 = multi-user.target (servers, no GUI), runlevel 5 = graphical.target, runlevel 6 = reboot.target. Switch with systemctl isolate multi-user.target (like telinit 3).
7Common Mistakes & Pro Tips
❌
Mistake 1 — Forgetting daemon-reload After editing any unit file, systemctl restart uses the OLD config. Always run sudo systemctl daemon-reload before restarting. Symptom: your changes appear to have no effect.
❌
Mistake 2 — Confusing start with enable Starting a service without enabling it means it won't return after a reboot. In production deploys, always use systemctl enable --now.
❌
Mistake 3 — Using restart when reload is sufficient restart drops all active connections. For web servers and load balancers, always prefer reload for config changes.
❌
Mistake 4 — Editing /lib/systemd/system/*.service directly Package upgrades will overwrite your changes. Use systemctl edit for persistent, upgrade-safe customization.
✅
Pro Tip 1 — Use -l --no-pager in scripts systemctl status nginx -l --no-pager — prevents truncation and disables interactive pager; essential for piping output or log capture in automation.
✅
Pro Tip 2 — Add Restart=on-failure to unit files Combined with RestartSec=5s, this makes services self-healing. If the ML model server crashes under load, systemd automatically recovers it within 5 seconds.
✅
Pro Tip 3 — Use --quiet for scripting systemctl is-active --quiet nginx && run_task — suppresses all output and only sets the exit code; perfect for if conditions and health check loops.
✅
Pro Tip 4 — systemctl --failed is your first debug command Immediately shows all broken services after a boot issue or failed deploy — no log sifting required. Make it a reflex.

service
COMMAND
Older service management command — SysV compatibility wrapper
1Command Overview

service is a wrapper script at /usr/sbin/service that provides a unified interface across different init systems — SysV init, Upstart, and systemd. On systemd-based systems, service commands are transparently redirected to systemctl.

It predates systemd and remains in use for compatibility with older scripts, Ansible targeting multiple distros, and legacy systems (CentOS 6, Ubuntu 14.04, Debian 7). On modern systems, always prefer systemctl directly — service cannot handle boot persistence, dependency trees, timers, or targets.

⚠️
Key Distinction service nginx start = systemctl start nginx on systemd systems. However, there is NO service enable equivalent — for boot persistence you must use systemctl enable.
2Syntax
BASH
service SERVICE_NAME COMMAND

  SERVICE_NAME — Name of the service (nginx, mysql, ssh)  — NO .service suffix!
  COMMAND      — start | stop | restart | reload | status

5Real-World Use Cases
  • Cross-distro Ansible: Ansible's service module automatically calls service on SysV systems and systemctl on systemd — write once, run everywhere.
  • Legacy servers: Organizations running CentOS 6 or Ubuntu 14.04 — service is the only option on these systems.
  • Docker images: Minimal containers use service as it requires fewer dependencies than a full systemd setup.
  • Global status audit: service --status-all gives an instant snapshot of all services on SysV systems — equivalent to systemctl list-units.
6Interview Questions & Answers
Q1What is the relationship between service and systemctl?
A:On systemd-based systems, service is a compatibility shim. When you run service nginx start, it detects that systemd is running and internally calls systemctl start nginx. The redirection is transparent. service cannot perform systemd-specific actions like enable, disable, mask, daemon-reload, or manage targets.
Q2On a SysV system (no systemd), how does service work?
A:On SysV systems, service NAME COMMAND executes the corresponding init script at /etc/init.d/NAME with the given command. For example, service nginx restart runs /etc/init.d/nginx restart. Boot persistence is managed separately via update-rc.d (Debian) or chkconfig (RHEL).
Q3Scenario: Writing an Ansible playbook for both CentOS 6 and CentOS 8. How do you handle service management?
A:Use Ansible's service module — it auto-detects the init system. Example: - name: Ensure nginx is running service: name: nginx state: started enabled: yes. Ansible handles the detection internally, calling service on SysV and systemctl on systemd — fully cross-distro compatible.
Q4What do the symbols in service --status-all output mean?
A:[ + ] = running, [ - ] = stopped, [ ? ] = status cannot be determined (the init script doesn't implement the status action). On systemd systems, this output is generated by querying systemctl for each unit. Less detailed than systemctl list-units but useful for a quick global snapshot.
7Common Mistakes & Pro Tips
❌
Mistake — Expecting boot persistence service start nginx starts it now but there is NO service enable. On modern systems, you must separately run systemctl enable nginx for boot persistence.
❌
Mistake — Mixing service and systemctl Without detecting which is available, mixing both in the same script can cause conflicts — especially during OS upgrades.
✅
Pro Tip — Detect the init system first if systemctl is-active --quiet nginx 2>/dev/null; then... This avoids calling the wrong tool and safely handles containers where systemd may not be running.
✅
Pro Tip — In interviews, show maturity Always mention you prefer systemctl on modern systems, but understand service for cross-distro compatibility and legacy environments. This signals production experience.

Final Interview Tip

Interviewers love asking: "A service is failing after deployment. Walk me through your troubleshooting steps."

Perfect answer: systemctl --failed → systemctl status <unit> -l → journalctl -u <unit> -b → systemctl cat <unit> → fix → daemon-reload → systemctl start <unit>.

Knowing this flow confidently, in order, makes a strong impression. It shows you've actually debugged services in production, not just memorized commands.
